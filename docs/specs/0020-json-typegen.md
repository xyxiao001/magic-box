# 0020 JSON Typegen

## 背景

从接口响应、配置文件或日志里拿到一段 JSON 后，最常见的下一步就是“生成类型定义”。对前端来说需要 TypeScript interface/type；对后端/全栈来说常用 schema（如 Zod）做校验与推断。一个本地的 JSON → TS/Zod 工具能显著减少重复劳动。

## 用户故事

作为前端/全栈开发者，我希望粘贴 JSON 后立即得到 TypeScript 类型或 Zod schema，并能选择风格配置（可选字段、readonly、命名等），从而更快写接口层与校验逻辑。

## 首版范围

- 输入：
  - 支持粘贴 JSON（对象或数组）
  - 支持格式化与校验（错误路径提示）
- 输出：
  - TypeScript：`interface` / `type`（二选一）
  - Zod：`z.object({...})`，并导出 `z.infer<>` 类型
- 推断规则（首版）：
  - 字符串/数字/布尔/null
  - 数组：同构数组与混合数组（混合数组用 union）
  - 对象：字段推断、嵌套对象生成子类型
  - 可选字段：当数组里多个对象字段不一致时，将缺失字段标为 optional
- 配置项（首版）：
  - 根类型名（默认 `Root`）
  - 生成 TS 的风格：interface 或 type
  - 生成 Zod 时是否启用 `.strict()`（默认不开）
  - 是否把 `null` 合并为 optional（默认否）
- 操作：
  - 一键复制输出
  - 一键下载为 `.ts`（可选）

## 不做什么（首版不含）

- 不做 OpenAPI 生成
- 不做 JSON Schema 全量导出（可作为下一阶段）
- 不做复杂字符串格式推断（uuid/email/date），只做基础类型

## 交互与页面

- 左侧：JSON 输入区 + 校验提示
- 右侧：tabs（TypeScript / Zod）+ 配置区 + 复制按钮

## 数据与状态

- 本地存储：保存最近一次输入与配置项
- 不上传内容，本地推断与生成

## 验收标准

- 输入合法 JSON：
  - 能生成可编译的 TS 类型
  - 能生成可运行的 Zod schema
- 输入非法 JSON：
  - 能给出清晰错误提示（包含行列或路径）
- 当数组里对象字段不一致时：
  - 生成 optional 字段符合预期
- 复制输出可用

## 风险与待确认问题

- 类型推断边界很多，首版以“可用 + 可解释”为优先
- 深层嵌套 JSON 可能生成大量类型，需要考虑输出可读性与命名策略
